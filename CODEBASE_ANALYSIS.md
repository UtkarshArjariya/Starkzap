# Starkzap — Codebase Analysis & Roadmap Reference

> Updated: 2026-03-17 | Branch: main | Status: clean

## 1. Product Summary

**Starkzap** (also referred to as "Dare Board") is a Starknet-based social challenge platform.

Core loop:
1. A **poster** creates a dare and escrows a token reward on-chain.
2. A **claimer** accepts the dare before the deadline.
3. The claimer submits public **proof** of completion.
4. The **community votes** approve/reject during a 24-hour window (minimum 3 votes required).
5. Anyone calls **finalize** — the contract pays the claimer on approval, or refunds the poster on rejection/expiry.
6. The poster can **cancel** an unclaimed dare and reclaim their escrowed reward.

The entire reward lifecycle is on-chain, transparent, and trustless.

**Production URL:** `https://dareboard.vercel.app`

### 3.2 Data Structures

#### `DareStatus` (enum)
```
Open → Claimed → Voting → Approved
                        ↘ Rejected
Open / Claimed → Expired (if deadline passes)
Open → Cancelled (poster cancels before claim)
```

| Variant      | Meaning                                      |
|--------------|----------------------------------------------|
| `Open`       | Posted, awaiting a claimer                   |
| `Claimed`    | Someone accepted; awaiting proof             |
| `Voting`     | Proof submitted; 24-hour community vote open |
| `Approved`   | Votes passed; reward released to claimer     |
| `Rejected`   | Votes failed; reward returned to poster      |
| `Expired`    | Deadline passed; escrow returned to poster   |
| `Cancelled`  | Poster cancelled; escrow returned            |

#### `Dare` (view struct, never stored)
| Field                | Type            | Description                          |
|----------------------|-----------------|--------------------------------------|
| `id`                 | `u64`           | Auto-incremented dare ID (1-indexed) |
| `poster`             | `ContractAddress` | Creator's wallet address           |
| `title`              | `felt252`        | Short title (max 31 ASCII chars)    |
| `description`        | `ByteArray`      | Full description                    |
| `reward_token`       | `ContractAddress` | ERC20 token address                |
| `reward_amount`      | `u256`           | Escrowed reward (18 decimals)       |
| `deadline`           | `u64`            | Unix timestamp                      |
| `claimer`            | `ContractAddress` | Who claimed it (zero if unclaimed) |
| `proof_url`          | `ByteArray`      | Proof link                          |
| `proof_description`  | `ByteArray`      | Proof context                       |
| `proof_submitted_at` | `u64`            | When proof was submitted            |
| `voting_end`         | `u64`            | `proof_submitted_at + 86400`        |
| `approve_votes`      | `u64`            | Running approve count               |
| `reject_votes`       | `u64`            | Running reject count                |
| `status`             | `DareStatus`     | Current state                       |

#### Storage
```cairo
Storage {
    dare_count: u64,                            // total dares ever created
    owner: ContractAddress,                     // deployer/owner
    dares: Map<u64, DareNode>,                  // dare_id → dare data
    has_voted: Map<(u64, ContractAddress), bool> // (dare_id, voter) → voted?
}
```

### 3.3 Public Interface (IDareBoard)

| Function           | State | Params                                                    | Returns | Notes |
|--------------------|-------|-----------------------------------------------------------|---------|-------|
| `create_dare`      | write | title, description, reward_token, reward_amount, deadline | u64     | Pulls ERC20 into escrow via `transfer_from`; deadline must be >1h from now |
| `claim_dare`       | write | dare_id                                                   | —       | Status must be Open; caller ≠ poster; block_ts < deadline |
| `submit_proof`     | write | dare_id, proof_url, proof_description                     | —       | Caller must be claimer; sets voting_end = now + 86400 |
| `cast_vote`        | write | dare_id, approve: bool                                    | —       | Status must be Voting; voter ≠ poster, ≠ claimer; one vote per address |
| `finalize_dare`    | write | dare_id                                                   | —       | Anyone can call; requires MIN_VOTES_TO_FINALIZE (3) for vote-based outcomes |
| `cancel_dare`      | write | dare_id                                                   | —       | Only poster; status must be Open; refunds escrowed reward |
| `get_dare`         | read  | dare_id                                                   | Dare    | Returns full view struct |
| `get_dare_count`   | read  | —                                                         | u64     | Total dares created |
| `has_voter_voted`  | read  | dare_id, voter                                            | bool    | Vote deduplication check |

### 3.5 Finalization Logic

```
IF status == Voting AND now >= voting_end:
    IF (approve_votes + reject_votes) < MIN_VOTES_TO_FINALIZE (3):
        assert(false, 'Not enough votes to finalize')
    IF approve_votes > reject_votes:
        transfer(reward, claimer)
        status = Approved
    ELSE:
        transfer(reward, poster)
        status = Rejected

ELSE IF (status == Claimed OR Open) AND now > deadline:
    transfer(reward, poster)
    status = Expired

ELSE:
    assert(false, 'Cannot finalize yet')
```

**Note:** Equal votes default to `Rejected` (poster refund). Minimum 3 total votes required.

### 3.7 Security Audit Status

**Completed:** All 6 ERC20 `transfer()` and `transfer_from()` calls now have `assert()` on return values to prevent silent failures with non-standard tokens (USDC/USDT).

### 3.8 Test Coverage

15 snfoundry tests in `contracts/tests/test_dare_board.cairo` with a MockERC20:
- Full dare lifecycle (create → claim → proof → vote → finalize)
- Access control checks (poster can't claim, non-claimer can't submit proof, etc.)
- Vote threshold enforcement (minimum 3 votes)
- Cancel dare + refund
- Expiry handling
- All 15 tests pass (`scarb test`)

### 3.9 Known Contract Limitations

1. **No tie-breaker** — equal votes → Rejected, which may surprise users.
2. **No dare edit** — once created, title/description/reward are immutable.
3. **Single claimer model** — only one person can claim at a time; no competition.
4. **Title felt252 limit** — max 31 ASCII chars; ByteArray description is unlimited.
5. **`owner` stored but unused** — constructor stores owner but no admin-only functions exist yet.

### 4.4 Wallet Support

Three wallet connection paths (in `WalletModal.tsx`):

1. **Extension wallets** (Argent X, Braavos) — via `@starknet-io/get-starknet-core` v4. User pays gas.
2. **Cartridge Controller** — via StarkZap SDK `connectCartridge()`. Social login, built-in paymaster (zero gas).
3. **Privy social login** (Email/Google/Apple) — partially implemented, not yet committed. Requires `@privy-io/react-auth` + server routes.

**Important:** Extension wallets CANNOT use AVNU paymaster for gasless tx — the paymaster requires co-signing before the user's wallet signs, but extensions sign internally and never expose the unsigned transaction.

Server-side finalize route uses deployer `Account` directly via `DEPLOYER_PRIVATE_KEY`.

## 10. Notable Bug Fixes

| Issue | Root Cause | Fix |
|-------|-----------|-----|
| "Invalid block_id" on feed load | Alchemy RPC v0_10 doesn't support `"pending"` block tag | Set `blockIdentifier: "latest"` in RpcProvider |
| Wallet rejects create_dare calldata | starknet.js `encodeShortString` uses `/./g` regex which skips `\n` chars, producing broken hex | Custom `encodeShortStringFixed()` + `stringToByteArrayCalldata()` using char-by-char iteration |
| Deploy script "Cannot convert undefined to a BigInt" | starknet.js v1 transactions deprecated on Sepolia, fee estimation broken | Rewrote deploy script to use `sncast` CLI instead of starknet.js Account |
| Generic "Transaction failed" errors | `decodeContractError` had generic fallback hiding real errors | Added felt252 hex decoding, quoted reason extraction, truncated raw error display |
| Unchecked ERC20 return values | `transfer()` / `transfer_from()` could silently fail with non-standard tokens | Added `assert()` on all 6 ERC20 calls in contract |
